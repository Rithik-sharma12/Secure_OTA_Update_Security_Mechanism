## What already exists

There are already systems and research that cover large parts of your idea:

- **Open‑source / commercial platforms**
  - OTA Hub: uses GitHub releases, CI/CD to build `.bin` firmware, tracks which device has which firmware, and supports optional code‑signing. [otahub.wordpress](https://otahub.wordpress.com)
  - Mongoose OS: IoT firmware framework with its own OTA update support and cloud integrations (AWS, Azure, etc.). [github](https://github.com/cesanta/mongoose-os)
  - Golioth, Notehub, etc.: have GitHub Actions to build firmware and automatically deploy OTA to devices, including staged rollouts and rollbacks. [youtube](https://www.youtube.com/watch?v=FZinwXH0O1M)

- **Research papers / standards**
  - “Secure Firmware Updates Over the Air in the IoT” (IETF/academic work) defines requirements like integrity, authenticity, rollback protection, fail‑safe backup image, etc. [summit.riot-os](http://summit.riot-os.org/2024/wp-content/uploads/sites/19/2024/09/01-3-Silvie_Schmidt_Miguel_Arcilla.pdf)
  - A 2022 paper proposes a “highly secure IoT firmware update mechanism” using blockchain to protect integrity, anonymity, and update records against tampering. [pmc.ncbi.nlm.nih](https://pmc.ncbi.nlm.nih.gov/articles/PMC8781858/)
  - Many surveys and guides summarize best practices: TLS‑secured channels, cryptographic signing, anti‑rollback, device authentication, etc. [stormotion](https://stormotion.io/blog/updating-iot-devices/)

So: **secure OTA + CI/CD + GitHub + integrity/MITM protection absolutely exists as a concept and in implementations.** [linkedin](https://www.linkedin.com/posts/nainaiu-rakhaine_you-can-update-firmware-version-in-your-activity-7362551146424516608-LYzU)

## Typical drawbacks in existing work

Most existing tools or papers don’t give a complete, practical “college‑level full‑stack” solution covering the whole lifecycle the way you’re imagining. Key gaps:

- **Fragmented lifecycle**
  - CI/CD builds and pushes binaries but doesn’t model update policies, device groups, or per‑device version history in a way you can easily study as a research problem. [dev.blues](https://dev.blues.io/blog/github-action-notehub-odfu/)
  - Many frameworks stop at “device downloads signed firmware over TLS” without deeper treatment of governance (who approves what, how rollbacks are handled, how compliance is audited). [ietf](https://www.ietf.org/slides/slides-iotsuws-secure-firmware-update-over-the-air-in-the-internet-of-things-focusing-on-flexibility-and-feasibility-00.pdf)

- **Security scope limitations**
  - A lot of implementations only ensure *integrity + authenticity* via signing, but don’t always enforce:
    - Anti‑rollback (prevent downgrading to vulnerable versions). [stormotion](https://stormotion.io/blog/updating-iot-devices/)
    - Fine‑grained device authorization (this device is allowed to get only these channels/branches).
    - Defense against key compromise (key rotation, revocation lists).
  - Some research (e.g., blockchain‑based updates) is heavy and not very practical for constrained ESP32/MCU environments. [pmc.ncbi.nlm.nih](https://pmc.ncbi.nlm.nih.gov/articles/PMC8781858/)

- **Developer‑friendly integration**
  - Existing CI/CD examples (LinkedIn posts, Golioth demo, GitHub Actions for firmware) are often **ad‑hoc**, not generic frameworks with clear abstractions and documentation. [github](http://github.com/9elements/firmware-action/actions)
  - Very few provide:
    - Simple, reproducible template repos (Arduino/ESP‑IDF) where a student can fork and immediately have end‑to‑end secure OTA from Git push to device.
    - Built‑in formal threat model and automated security checks (e.g., rejecting unsigned or downgraded builds automatically).

- **Verification and observability**
  - Many systems do not expose verifiable proofs to the device or to an auditor that:
    - “This binary exactly came from commit X and passed security tests Y.”
    - “This device was never allowed to run unapproved firmware.”
  - Logging is often centralized and not tamper‑evident; you can’t easily show a security auditor that no MITM or rollback happened. [stormotion](https://stormotion.io/blog/updating-iot-devices/)

These gaps are where you can contribute something meaningfully “novel enough” for a college project.

## Core Architecture

Your system has three layers:
- **Blockchain layer**: Smart contract stores firmware manifest (version, hash, IPFS/S3 link, device ID, timestamp). Devices query for updates via RPC (no need to run full node). [github](https://github.com/sfl0r3nz05/DistributedBchFWArchIoT)
- **Off-chain storage**: Firmware binaries in IPFS (decentralized, cheap) or GitHub Releases/S3. Blockchain only holds pointers + proofs. [github](https://github.com/paltrickontpb/BlockOTA)
- **Device layer**: ESP32 polls blockchain (via proxy server or lightweight lib), verifies hash/signature, downloads if valid, applies OTA with rollback.

Flow:
1. Manufacturer pushes firmware to GitHub → CI signs it → uploads hash/version to smart contract.
2. Device queries contract: "latest version > mine for my device type?" → gets IPFS hash.
3. Downloads, verifies integrity/authenticity, updates.

This prevents MITM (crypto proofs), rollbacks (monotonic versions), and tampering (immutable ledger). [dl.acm](https://dl.acm.org/doi/10.1145/3312614.3312649)

## Lightweight Implementation Steps

Target Ethereum Sepolia testnet (free, RPC access via Infura/Alchemy). Use Solidity for contract, ESP32 Arduino/ESP-IDF for device.

### 1. Smart Contract (Solidity)
Deploy a simple contract for firmware registry. Key functions:
- `publishUpdate(deviceType, version, firmwareHash, ipfsCid, signature)` – only owner calls.
- `getLatestUpdate(deviceType)` – returns latest valid version/hash for device group.

Example skeleton (adapt from BlockOTA/paper pseudocode):
```solidity
contract IoTUpdateRegistry {
    struct Update { uint version; bytes32 hash; string ipfsCid; address publisher; }
    mapping(string => Update[]) public updates; // deviceType -> list of updates
    
    function publishUpdate(string memory deviceType, uint version, bytes32 hash, string memory ipfsCid) public {
        // Verify signature, monotonic version, etc.
        updates[deviceType].push(Update(version, hash, ipfsCid, msg.sender));
    }
    
    function getLatestUpdate(string memory deviceType) public view returns (uint, bytes32, string memory) {
        Update[] memory list = updates[deviceType];
        return (list[list.length-1].version, list[list.length-1].hash, list[list.length-1].ipfsCid);
    }
}
```
- Deploy with Remix/Hardhat. Cost: ~$0.01 per tx on testnet.
- GitHub Actions: After build, use ethers.js to call `publishUpdate`. [github](https://github.com/paltrickontpb/BlockOTA)

### 2. Firmware Signing & Hashing (CI/CD)
- In GitHub Actions: Build ESP32 `.bin`, compute SHA256 hash, sign with ECDSA (secp256k1, matches Ethereum).
- Store signed manifest: `{version: "1.2", hash: "0x...", sig: "0x...", ipfsCid: "..."}`.
- Pin binary to IPFS (via Pinata or Infura gateway, free tier). [github](https://github.com/paltrickontpb/BlockOTA)

### 3. ESP32 Device Code (Arduino/ESP-IDF)
Devices can't run full blockchain nodes (RAM/CPU too low), so use HTTP RPC proxy:
- Libs: `ArduinoJson`, `HTTPClient`, `esp32-http-ota`.
- Poll every X hours: Query Alchemy RPC for contract data.
- Verify: recompute hash, check signature with public key.

Pseudocode:
```cpp
#include <WiFi.h>
#include <HTTPClient.h>
// Add secp256k1 lib for sig verify (tiny-ecdh or similar)

void checkUpdate() {
  HTTPClient http;
  http.begin("https://sepolia.infura.io/v3/YOUR_KEY"); // RPC endpoint
  http.POST("eth_call: getLatestUpdate('esp32-type')"); // Parse JSON response
  String latestHash = parseHash(response);
  if (latestHash != currentHash && verifySig(latestHash, sig, pubKey)) {
    http.begin("https://ipfs.io/ipfs/" + ipfsCid);
    downloadAndOTA("/tmp/firmware.bin"); // Use ESP OTA lib with rollback
  }

```
- Security: Use HTTPS, store pubkey in secure NVS/HSM sim (flash fuses). [pmc.ncbi.nlm.nih](https://pmc.ncbi.nlm.nih.gov/articles/PMC8781858/)
- Dual partitions for safe OTA (built-in ESP-IDF). [sunfounder](https://www.sunfounder.com/blogs/news/esp32-ota-updates-a-complete-guide-to-arduinoota-and-elegantota-firmware-upgrades)

## Drawbacks & Fixes for College Project
- **Gas/cost**: Testnet free; mainnet ~$0.10/tx. Fix: Batch updates.
- **Latency**: RPC polling adds 1-2s. Fix: MQTT pubsub via blockchain events (TheGraph indexer).
- **IoT constraints**: No full node. Fix: Proxy server (RPi/AWS Lambda). [github](https://github.com/sfl0r3nz05/DistributedBchFWArchIoT)
- **Scalability**: Ethereum slow for 1M devices. Fix: Layer2 (Polygon) or Hyperledger for permissioned net.
