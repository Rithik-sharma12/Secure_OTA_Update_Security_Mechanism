Existing Projects & Research Papers
1. SEMAR CI/CD-Enabled OTA (2025 — Most Relevant Paper)
This paper proposes an automated OTA firmware update mechanism integrated with CI/CD to ensure trusted firmware origins. It addresses version rollback, modified firmware files, and potential MITM attacks. The workflow starts from the developer side, goes through a CI/CD runner for automated build and signing, followed by firmware delivery to devices, where signature verification is performed using a public key stored on the device. If any verification step fails, the update process is terminated. Stack Overflow
This is the closest paper to your idea. However, it relies on SEMAR, their own custom IoT application server from prior studies — not GitHub as the free backbone. It also doesn't address the full version control lifecycle with semantic versioning or rollback policies.

2. PARIOT — Anti-Repackaging for IoT Firmware Integrity
PARIOT ensures firmware update integrity through a self-protection mechanism that does not require signing keys, internet connection, secure storage technologies, or external trusted parties. It is agnostic to the firmware delivery model and can be layered on top of existing solutions with negligible overhead. Stormotion
This is novel in that it avoids key management complexity, but it does not integrate CI/CD at all, and has no deployment pipeline or GitHub-based automation.

3. IMUP — Integrity-Centric Modular Update Platform (2025)
IMUP addresses two challenges: constructing a trustworthy cross-module integrity chain, and scaling update performance under mass customisation. It integrates per-module chameleon hashing to enable verifiable integrity across customised firmware compositions, and server-side proof-of-work offloading. CEUR-WS.org
This is cutting-edge but focuses on modular firmware (not monolithic), and is complex — it also does not use GitHub as the distribution channel.

4. Blockchain-Based OTA (2019, Blockchain + ESP8266)
This system uses a blockchain framework with smart contracts to protect the integrity of the firmware update process and was evaluated for scalability and response to DoS and MITM attacks. Bugprove
Blockchain adds decentralization, but it's computationally expensive, requires gas fees on most chains, and is overkill for a college-scale IoT system. Constrained devices (ESP32, Arduino) cannot run blockchain clients natively.

5. Mender, JFrog Connect, AWS IoT OTA, Azure Device Update (Commercial Tools)
Mender works with real-time operating systems such as Zephyr and all Linux-based OSes, is cross-compatible and API-driven, and can be integrated into CI/CD pipelines. It follows a security-by-design principle ensuring each step is verified and authorized. PubMed Central
Azure Device Update for IoT Hub is an end-to-end platform supporting A/B updates with seamless rollback, subscription and role-based access controls, and at-a-glance compliance views across heterogeneous device fleets. Springer
These are polished and production-grade, but they are paid, vendor-locked, and cloud-dependent (AWS, Azure, JFrog infrastructure). A student or small team cannot freely replicate them.

6. Husarnet + GitHub Actions (Closest Free Alternative)
Husarnet Action allows firmware to be sent directly to an edge device without the need for complicated configuring with an intermediary IoT cloud. The code and remote firmware update setup are covered by version control in a single GitHub repository, functioning as a true Infrastructure as Code approach to handling IoT fleets. ResearchGate
This is free and uses GitHub — but it focuses on deployment topology (peer-to-peer VPN), not on security properties like integrity checking, signature verification, or MITM prevention as first-class features.

The Core Drawbacks of ALL Existing Work
Here is the critical gap analysis that makes your project novel:
Fragmentation of the security lifecycle. Existing OTA update mechanisms primarily concentrate on optimizing firmware dissemination efficiency, communication reliability, or bandwidth reduction. Security mechanisms are often treated as complementary components rather than integral elements of the entire firmware lifecycle. Most studies isolate the distribution phase from earlier development stages, leaving firmware build automation, secure signing processes, and controlled release management largely unaddressed. Stack Overflow
Vendor lock-in. Every mature solution (AWS OTA, Azure DU, Mender, JFrog) requires you to use their cloud infrastructure. There is no free, self-contained, GitHub-native pipeline that handles the full security lifecycle from commit to verified device installation.
Rollback is neglected. Most papers address the forward update path but not safe, cryptographically-verified rollback to a known-good version when a bad update is detected.
Constrained-device confidentiality is weak. Despite improvements in OTA mechanisms, device developers are generally reluctant to provide security patching as a continual maintenance service. Devices may be running obsolete firmware several years old with widely acknowledged flaws. Where OTA updates and encryption mechanisms are available, the network protocols also need to be tested for security compliance. GitHub
No update audit trail. None of the free/open implementations include a tamper-evident log of what was pushed, when, by whom, and what the device acknowledged — which is critical for compliance and forensics.
Over 50% of update-related CVEs reported between 2020 and 2024 originate from customisation interfaces, and current defences fail to keep pace. CEUR-WS.org This means the threat is growing but the free tooling is not.

What Makes YOUR Project Novel — The Unique Angle to Target
Given all of the above, here is how you carve out genuinely new ground:
Your project can be the first fully free, open-source, GitHub-Actions-native, end-to-end secure IoT OTA lifecycle system that combines all of the following in one coherent framework — no paid cloud, no vendor lock-in:
Stage 1 — Source (GitHub): Developer pushes firmware code to GitHub. A GitHub Actions workflow automatically builds, signs with a private key (stored as a GitHub Secret), generates a SHA-256 hash manifest, creates a versioned release using semantic versioning (v1.0.0, v1.0.1...), and attaches the signed binary + manifest to a GitHub Release.
Stage 2 — Distribution (Integrity + Confidentiality): The firmware binary is encrypted (AES-256) before being hosted. The device downloads it over HTTPS, preventing MITM by certificate pinning. The manifest (hash + signature) is fetched separately, ensuring even if the binary is intercepted it cannot be decrypted or forged.
Stage 3 — Device Verification (Trust Chain): The device holds only the developer's public key baked in at compile time. It verifies the digital signature of the manifest, then recomputes the SHA-256 hash of the downloaded binary and compares it against the signed manifest. If anything fails, it rejects the update entirely and triggers rollback.
Stage 4 — Version Control + Rollback: You maintain a version ledger (a simple JSON file in the GitHub repo) tracking current version per device group, last known-good version, and update history. If a device reports a failed boot after an update, the pipeline automatically triggers a rollback release to the last known-good version.
Stage 5 — Audit Log: Every update event (release pushed, device acknowledged, device rejected, rollback triggered) is written to a tamper-evident append-only log using HMAC chaining — a lightweight alternative to blockchain.
This combination — free infrastructure + full security lifecycle + audit trail + auto-rollback + GitHub-native — does not exist as a single published open-source project today. Each component exists, but not unified like this.
