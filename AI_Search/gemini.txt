### The Gaps & Drawbacks (Your "Problem Statement")

To make your project stand out, address these common failures in existing systems:

1. **The "Brick" Risk:** Most DIY updates don't have **A/B Partitioning**. If the update fails mid-way, the device dies.
2. **Lack of Manifests:** Many systems send just the binary. A secure system needs a **Signed Manifest** (metadata containing version, size, hash, and hardware compatibility).
3. **Key Management:** Often, private keys are stored insecurely in the GitHub repo secrets or on the dev machine.
4. **No Mutual Authentication:** The device trusts the server, but the server doesn't always verify the device, leading to "ghost" devices receiving updates.

### Your Proposed Lifecycle (The "Unique" Path)

To ensure your project is "novel," I recommend building this specific flow:

1. **Code Commit:** You push code to GitHub.
2. **GitHub Action (The "Signer"):** * Compiles the code.
* Generates a **Digital Signature** using a private key (stored in GitHub Secrets).
* Creates a `manifest.json` with the hash and version.


3. **Deployment:** The Action pushes the signed binary to a "Release" or a secure bucket.
4. **Device Check:** The IoT device (e.g., ESP32) checks for a new version using **TLS 1.3**.
5. **The "Secure Gate":** * **Integrity:** Device checks the SHA-256 hash.
* **Authenticity:** Device uses a **Public Key** (pre-flashed) to verify the signature.
* **Anti-Rollback:** Device rejects the update if the version number is lower than the current one.


6. **Dual-Bank Update:** The device writes to "Partition B" while running on "Partition A." It only switches if the update is 100% verified.
